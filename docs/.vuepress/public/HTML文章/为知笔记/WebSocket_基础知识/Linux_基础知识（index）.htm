2、常见开发问题



2.1、curl


问题：curl 下载阿里云 maven 仓库上的依赖包为 0 kb

问题描述：
使用 url 下载阿里云 maven 仓库上的依赖包时，浏览器上正常下载，curl 上下载为 0 kb。
下载地址如下：
Cordova 使用的地址（Aliyun）：
https://maven.aliyun.com
/repository/public/com/android/databinding/baseLibrary/3.0.1/baseLibrary-3.0.1.jar

Ali 仓库给出的地址（Aliyun OSS）：
https://archiva-maven-storage-prod.oss-cn-beijing.aliyuncs.com
/repository/google/com/android/databinding/baseLibrary/3.0.1/baseLibrary-3.0.1.jar
?spm=a2c6h.24085988.0.0.101c3054iEhKvu
&Expires=1638764703
&OSSAccessKeyId=LTAIfU51SusnnfCC
&Signature=MrsFYN8vp4Dq3XZiR%2BPU6MTO0U4%3D

其中，Cordova 用的地址只能在浏览器中下载，而阿里仓库给的地址都可以。
为什么 Cordova 给的依赖包地址只能在浏览器中下载，而不能在 curl 中下载呢？


问题原因：

1. 原因分析：

首先要确认是不是 curl 版本问题。
这就需要测试几种 Shell 工具自带的 curl  下载文件的情况：
git bash：
Cordova 使用的地址：0 kb
Ali 仓库给出的地址：成功
powershell ：
Cordova 使用的地址：成功
Ali 仓库给出的地址：0 kb
cmd（第三方）：
Cordova 使用的地址：0 kb
Ali 仓库给出的地址：成功

版本说明：以上 curl 的版本都在 7.0+，是比较新的。

网查一堆资料，都说是 https 引起的，让开启“允许以不安全的方式访问 https”。然并卵。 
无奈只能仔细看日志。

curl 请求过程的日志中这样一句：
1
 
1
< HTTP/1.1 302 Found
它表示 curl 收到了响应，响应的状态码是 302。
302 ？应该是 https 的原因被服务器拒绝了吧？其实不然，这是正常响应。
HTTP 响应状态码 3XX 表示 redirection，重定向，重定向的 URL 记录在返回的报文中的 Location 字段。

重定向？阿里仓库 url 重定向了？为什么？先不管为什么，但这应该就是 curl 下不了文件的原因了，先看看 curl 有没有允许重定向的方法。
找到 curl 允许重定向的方法后一试，OK，问题解决。



2. 分析结果：
问题的原因不是 https 出于安全拦截了 curl 的下载，而是阿里云把旧 URL 给重定向了，要下载的文件在重定向的新 URL 上， 但 curl 却默认不允许重定向，拿到响应结果就结束了，就没跳转到新 URL 上下载。而浏览器是默认是允许的，于是就跳转到新 URL 上下载了，只是因为浏览器地址栏没有显示新地址，所以我们察觉不到重定向了，不过从浏览器后台是可以看到的。



问题解法：
允许 curl 接受服务器的重定向指令。# 强制无证书访问 https、强制 http1.1、允许重定向
2
 
1
# 强制无证书访问 https、强制 http1.1、允许重定向
2
curl https://maven.aliyun.com/repository/public/com/android/databinding/baseLibrary/3.0.1/baseLibrary-3.0.1.jar -O --verbose --location



解题思路：
这个问题之所以能够解决，不是搜索引擎给了我最终答案，而是错误日志。
我先从错误日志中看到 http 协议由 1.1 改为了 2.0，于是改 http 协议为 1.1，问题依旧存在。
我又从错误日志中看到 Http 302，以为是被服务器拒绝的结果，但处于好奇，还是搜了一下，是关于重定向的，这时我突然就明白了一切。

Tips：
curl 默认使用 http 1.1，如果服务器支持 http 2 则使用 http2。

2.2、编码


问题：Linux 解压 tar.gz 文件名中文乱码

1、问题描述：
Windwos 10 上打包网页静态资源文件到云服务器上发布，打的压缩包是 tar.gz 的，解压后，用 UTF-8 编码的 SecureCRT、FlashFXP 看到的名字都是乱码，导致这些资源文件无法访问。
当前 Linux 是阿里云的 CentOS 8.2。

2、问题原因：
并非 Linux 不支持中文名称的文件和目录，而是 tar.gz 不保存字符集，而 Windows 上是 GBK，Linux 上是 UTF-8，因此乱码

3、问题解法
在 Linux 把解压后的文件用 convmv 从 GBK 转为 UTF-8 即可。如：
1
 
1
convmv -f gbk -t utf-8 -r --notest ./*
