设置代理



为 Git 设置代理：


命令行执行如下命令： 
 
1
# http 代理
2
git config --global http.proxy 'http://127.0.0.1:1080'
3
git config --global https.proxy 'http://127.0.0.1:1080'
4
# socks5 代理
5
git config --global http.proxy 'socks5://127.0.0.1:1080'
6
git config --global https.proxy 'socks5://127.0.0.1:1080'
7
# 取消代理
8
git config --global --unset http.proxy
9
git config --global --unset https.proxy
设置成功之后配置文件“ ~/.gitconfig”会添加该配置项。当然，也可以直接在该配置文件中添加。





附：为执行的各种 exe 程序设置代理：
执行的 exe 有 npm、yarn、cordova、curl 等等。


1. 用环境变量代理的 exe
通过读取“http_proxy”这样的环境变量走代理的 exe 有很多，如 curl，还有大部分的 Linux 命令行工具。
常用于代理的环境变量有 http_proxy、https_proxy、all_proxy，有的可能要大写。

● 在 Git Bash 中
 
1
# 在 Git Bash 中执行或在配置文件中添加如下命令
2
export http_proxy="http://127.0.0.1:1080/pac?auth=**************************************"
3
export https_proxy=$http_proxy
4
export HTTP_PROXY="http://127.0.0.1:1080/pac?auth=**************************************"
5
export HTTPS_PROXY=$HTTP_PROXY
配置文件有：“Git_Home/etc/bash.bashrc“、.“User_Home/bash_profile"

●  在 CMD 中
 
1
# 在 CMD 中执行如下命令或者在系统设置中添加如下环境变量
2
set http_proxy="http://127.0.0.1:1080/pac?auth=**************************************"
3
set https_proxy=%http_proxy%
4
set HTTP_PROXY="http://127.0.0.1:1080/pac?auth=**************************************"
5
set HTTPS_PROXY=%HTTP_PROXY%

●  在 PowerShell 中
 
1
# 在 PowerShell 中执行如下命令或者在系统设置中添加如下环境变量
2
env:http_proxy="http://127.0.0.1:1080/pac?auth=**************************************"
3
env:https_proxy="http://127.0.0.1:1080/pac?auth=**************************************"
4
env:HTTP_PROXY="http://127.0.0.1:1080/pac?auth=**************************************"
5
env:HTTPS_PROXY="http://127.0.0.1:1080/pac?auth=**************************************"
Tips：
最好不要用 PowerShell，用 CMD。 PowerShell 内部设定了很多东西，有时候会不走你用环境变量设置的代理。




2. 拥有独立代理配置的 exe
拥有独立代理配置的 exe 有 curl、npm、yarn、cordova 等等，他们的配置方法各不相同。


● curl
执行如下命令：
 
1
# 临时走代理
2
curl -x socks5://127.0.0.1:1079 http://www.google.com.hk         #  -x 等同于 --proxy
3
curl -x http://127.0.0.1:1080 http://www.google.com.hk
4
# 临时不走代理
5
curl --noproxy "*" http://www.google.com.hk

配置文件“~/.curlrc”（没有就新建）添加如下内容，任选其一：
 
1
# -- http
2
proxy=http:/127.0.0.1:1080
3
# -- socks5
4
proxy=socks5:/127.0.0.1:1079
5
# -- socks5
6
socks5=127.0.0.1:1024

附：检查 curl 有没有走代理
 
1
# curl 的 -v 参数会输出请求过程的路由信息，以便确定请求有没有走代理
2
curl -v ...
3
# 查看与代理端口相链接的端口
4
netstat -nat | grep 1080
5
# 查看端口相关的进程
6
lsof -i :1080


● npm、yarn：
执行如下命令：
 
1
# npm 和 yarn 设置代理的命令参数是一模一样的
2
# -- 设置代理
3
npm config set proxy http://username:password@server:port
4
npm config set https-proxy http://username:password@server:port
5
yarn config set proxy http://username:password@server:port
6
yarn config set https-proxy http://username:password@server:port
7
# -- 删除代理
8
npm config delete proxy
9
npm config delete https-proxy
10
yarn config delete proxy
11
yarn config delete https-proxy

Tips：
nrm 设置的是远程仓库，不是代理。
cordova 使用的是 npm 的仓库。


● cordova：
修改配置文件，添加如下配置项：
 
1
# 在build.gradle 同级目录下创建 gradle.properties 配置文件来设置代理服务器
2
# -- http
3
systemProp.http.proxyHost=127.0.0.1
4
systemProp.http.proxyPort=8000
5
# -- https
6
systemProp.https.proxyHost=127.0.0.1
7
systemProp.https.proxyPort=8000
8
# -- 不走代理的 IP
9
systemProp.http.nonProxyHosts=10.*|localhost

Git Bash


环境变量
Windows  的环境变量会成为 Git Bash 的环境变量，但 Git Bash 的环境变量不是 Windows 的环境变量。二者有引用关系，但作用域没有交叉。

机制：
Bash 会读取当前操作系统的环境变量存储到自己的环境的变量里，当你在 Bash 或 Bash 的配置文件中执行“ export JAVA_HOME=ccc ”时候，作用 是修改 Bash 自己的变量，而不是操作系统的环境变量。
要搞清楚一点，环境变量也是变量，只不过作用域比较大，会被很多程序（比如操作系统）用来做些事情，所以才称之为环境变量，它并不是不同于变量的特殊的存在。



设置：
命令行执行或配置文件中添加如下命令： 
 
1
export PATH="/d/Program Files/Git/bin:$PATH"     # 目录之间使用冒号 : 分隔
2
export HTTP_PROXY="http://127.0.0.1:1080"
配置文件有：bash.bashrc、.bash_profile（~/.bash_profile，如果没有就创建）
export 的作用：
    export 的作用是临时导出一个变量，以便变量能够被子 shell 读取，但是该变量不能被兄弟 shell 读取
 
1
export      # 导出某个变量，导出的变量只能被 git bash 的子 shell 读取，不能被兄弟 shell 读取
2
export -n   # 取消导出某个变量。
    如果要 git bash 的所有 shell 都能读取到某个变量，需要在配置文件中设置。

读取：
 
1
# 用 $ 读取变量
2
$PATH
3
$HTTP_PROXY
打印：
 
1
# 用 echo 打印读取的变量
2
echo $PATH
3
echo $HTTP_PROXY
4
printf $PATH  # 反对使用，的等同于 C 的 printf，限制多，正常显示路径很难
查看：
 
1
# 显示当前环境所有环境变量，包括 export 定义的，三个任选其一即可
2
env         
3
printenv
4
export


Tips：
printf 打印的路劲里之所会缺少一些东西，不仅仅是因为 \ 是特殊字符，还因为会有 \s  这样的特殊标识符。它跟 C 的 printf 一样。
虽然 printf 对斜杠敏感，但并不代表其他程序使用会有问题，比如 Cordova、Gradle。而且也不代表其他打印方法也会有问题，比如 echo。建议用 echo 打印来检查路径。

附：如何检验一个程序是否以管理员身份运行
任务管理器中查看进程信息中的用户名，如果用户名是管理员的用户名，如 Administrator，那就代表是以管理员身份运行的。如下图“图 - 任务管理器 Git Bash 进程信息”。






常见问题：

问题：Git Bash 获取不到修改或更新后的 Windows 环境变量

原因：
这是因为 Git Bash Here 打开 Git Bash 时，是依托于某个程序（如 360 桌面）打开的，类似打开了一个子 Shell。这种打开方式并没有阻止 Git Bash 以管理员身份打开，从任务管理器中的进程用户信息可以看出 Git Bash 确实是以管理员身份打开的。
那既然都以管理员身份打开，为什么还获取不到更新的环境变量？难道是只能获取用户变量，而不能获取系统变量？
不是的用户变量和系统变量不冲突，既然都是管理员身份了那么都能获取到。这个问题的关键就在于 Git Bash 是依托于那个程序打开的。正因为 Git Bash 是依托于那个程序打开的，所以 Git Bash 使用的环境就是那个依托程序的环境，而那个依托程序没重启过，环境也就没有变过，环境变量自然就不会变。这种情况下无论打开多少次 Git Bash ，拿到的环境变量都不会变。


解法：

以下解法任选其一：
  • 关闭依托程序（如 360 桌面）或者不要在依托程序上 Git Bash Here。
  • 在任意打开的文件夹里 Git Bash Here；
  • 不用 Git Bash Here，直接在打开的文件夹的地址栏执行“git-bash”，等同于当前文件夹打开 Git Bash； 
  • 不用 Git Bash Here，直接运行里执行 “git-bash” 打开 Git Bash，再 cd 到目标文件夹；


检验 Git Bash 是否拿到最新环境变量的妙招：
要看一个 Git Bash 是否拿到最新的环境变量，虽然可以通过打印环境变量来检验，但是这样效率不高。现在发现了一个高效直接的方法：
重启打开一个 Git Bash，从任务管理器中看该 Git Bash 的进程有没有作业对象 ID，如果有，则表示它拿到了最新的，如果没有则没拿到最新的，拿的是依托程序（父程序）的。
